\section{Introduction}

The goal of this project is to develop a software system for modifying the duration of selected segments of a speech signal. The framework is designed to support user-specified edits, through either a scaling factor, or a target duration, for any given segment. In addition, the system supports multiple edits within a single audio file, and is designed to perform duration modification while preserving speech characteristics as well as possible.

To this end, a processing pipeline was designed that loads an input audio signal, parses one or more user-requested edits, applies a duration modification to the selected segments, and writes the resulting output signal. This structure separates the speech-processing algorithm from the surrounding application logic, making the system more extensible and understandable for new users.

For the core duration-modification stage, the Time-Domain Pitch-Synchronous Overlap and Add (TD-PSOLA) method was chosen. PSOLA is a speech processing technique for time-scale modification, and was chosen for this project due to its ability to allow changes in segment duration, while attempting to preserve speech characteristics.

\section{System Overview}

This project was implemented in Python using a modular design to isolate responsibilities across each stage of the data processing pipeline. At a high level, the system takes an input speech waveform, parses the user-specified duration edits, applies duration modification to the selected segments via the PSOLA algorithm, and reconstructs a final edited output waveform. This separation of responsibilities between file I/O, user-input parsing, and the PSOLA algorithm was identified early on as a key architectural choice, allowing for ease of testing, debugging, and extension.

The system is composed of the following primary modules:

\begin{itemize}
  \item \textbf{Audio I/O:} Loads input WAV files and writes processed output audio files.
  
  \item \textbf{Edit Specification Interface:} Handles the parsing of user-specified edits, including start/stop time, and the choice of either a scaling factor or duration specification.
  
  \item \textbf{Segment Extraction:} Converts time-based edit definitions into sample indices, and extracts the corresponding waveform segments to be passed to the duration modification algorithm.
  
  \item \textbf{Duration-Scaling Interface:} Provides the interface between the general editing framework and the duration-modification algorithm, and applies PSOLA time-scale modification to the received speech segment.
  
  \item \textbf{Stitching/Reconstruction:} Re-inserts the modified segment back into the original input waveform, and preserves the unedited regions of the data.
  
  \item \textbf{Output Generation:} Produces the final edited waveform.
\end{itemize}

In addition to the core framework, an evaluation script was developed for post-processing and review. This script provides valuable outputs for analysis, such as waveform and spectrogram comparisons between the original and modified signals. These visual metrics help support an overall assessment of system performance, aside from a simple perceptual assessment.

\section{Software Architecture and Design}

This section provides a more detailed discussion of the software engineering decisions behind the implementation of each module. In addition to the core DSP objective, the project required a software framework and end-to-end pipeline that could support a collection of user-configurable edits. For ease of testing and understanding, the system was designed as a pipeline of isolated modules, each with clearly defined responsibilities and interfaces.

At the application level, a main layer, found in \texttt{main.py}, coordinates the flow between modules. It loads the input waveform, parses and validates edits, extracts the relevant segment, invokes the duration-scaling interface, reconstructs the waveform, and writes the edited output. This file keeps module interactions explicit, while avoiding entanglements between modules of distinct concerns.

\subsection{Audio I/O}

Audio input and output functionality was implemented in \texttt{audio\_io.py}. This module is responsible for reading input \texttt{.wav} files into an internal array representation, and writing the final processed waveform back to the user. Separating file handling from the remainder of the processing pipeline simplified both debugging and testing, since waveform manipulation could be developed independently of the details of file parsing.

The first responsibility of this file is to load a \texttt{.wav} file, and perform extensive validation to guard against invalid inputs and common edge cases. These checks include file existence, invalid or unsupported file contents, empty audio data, and invalid sample rate conditions. By concentrating such validation in a single module, the remainder of the pipeline can operate under ideal assumptions regarding the structure and validity of the loaded signal, namely that the data is structured as the remainder of the pipeline expects.

Secondly, the module handles writing the processed waveform to a \texttt{.wav} file. 

\subsection{Edit Specification Interface}

The edit specification interface, largely provided through \texttt{interface.py}, is responsible for determining and validating user-requested duration modifications. Each requested edit specifies a time interval within the original signal, with either a duration scaling factor or a target output duration. Alternatively, a \texttt{.csv} input may be accepted at this layer for multi-segment editing.

A key design goal of this interface was flexibility. This flexibility was crucial, due to the requirement that the system be designed to support both single-edit and multi-edit specifications: By accounting for all control paths, this interface allows a user to either modify a single segment directly from the CLI, or to provide multiple requested edits through a \texttt{.csv} input.

Validation was also an important responsibility of this module. Since downstream data processing assumes well-behaved edit formats, the editing interface performs error-handling checks such as interval consistency, nonnegative timing, and detection of overlapping edit regions in multi-edit mode. This provides immediate handling of invalid user-input, and prevents bad data from propagating further into the pipeline, thus improving the general robustness of the system.

\subsection{Segment Extraction}

Segment extraction was implemented in \texttt{segment.py}. This module is responsible for converting user-specified time intervals (provided by the edit interface) into the corresponding sample-index ranges, and extracting the selected waveform segment from the full audio input. The extracted segment can then be passed directly to the duration-modification module for processing.

This separation was necessary because it isolates the time-to-sample conversion logic from both the user interface layer and the signal-processing algorithm. As a result, it reduces coupling and makes the duration-modification module easier to test independently, since that module can operate on a clean one-dimensional sample array rather than needing to manage edit parsing or file-level concerns.

The segment extraction stage also includes several validation checks, including checks for negative times, invalid interval definitions, and empty extracted sample ranges. The output of this module is a one-dimensional array of samples, together with the location information needed for reinsertion into the original waveform.

\subsection{Duration-Scaling Interface}

The duration-scaling interface is the point at which the general editing framework connects to the core speech-processing algorithm. Its purpose is to accept an extracted waveform segment together with the requested edit parameters, apply a duration modification, and return a modified segment suitable for re-insertion into the original signal.

Section 5 below will go into significant detail regarding the structure and approach of the PSOLA algorithm.

\subsection{Stitching/Reconstruction}

Stitching and reconstruction are performed after the duration-modification stage returns a modified waveform segment. This functionality was implemented in \texttt{stitch.py}. The purpose of this module is to replace the original user-specified segment with the modified segment, while preserving and retaining the portions of the waveform that were not marked for editing.

\subsection{Output Generation}

Final output generation consists primarily of producing the edited waveform in \texttt{.wav} format. The edited audio output is generated through the same audio I/O layer used for file writing.

Additionally, supporting scripts were used to generate output plots and metrics such as waveform comparisons and spectrogram comparisons for qualitative and quantitative assessment. This script can be found as \texttt{evaluate.py}. Although this evaluation tool is not part of the core editing pipeline itself, it plays an important role by providing a mechanism by which we may assess the effectiveness of the duration-modification method.

\section{Configuration and Usage}

The project was designed to support straightforward CLI use for both single-edit and multi-edit duration modifications. The implementation assumes a mono \texttt{.wav} input file and accepts user-specified edit parameters that determine which portion of the waveform should be modified, and how the duration change should be applied.

\subsection{Environment and Dependencies}

The recommended execution environment for the project is Python 3.12 or later. The primary dependencies used by the framework are \texttt{numpy}, \texttt{scipy}, \texttt{matplotlib}, and \texttt{pytest}.

Dependencies may be installed with:

\begin{verbatim}
pip install numpy scipy matplotlib pytest
\end{verbatim}

\subsection{Input Assumptions}

The system operates on mono \texttt{.wav} speech files. Each requested edit is specified by a start time \(t_1\) and end time \(t_2\), together with either a duration scaling factor or a target output duration. These time-domain edit requests are converted internally into sample-index ranges for segment extraction and algorithmic processing.

The framework supports two modes of operation:
\begin{itemize}
    \item \textbf{Single-edit mode}: One speech segment is modified.
    \item \textbf{Multi-edit mode}: Multiple edits are provided, through a CSV specification file.
\end{itemize}

\subsection{Single-Edit Usage}

Single-edit mode is used for modifying one specified region of the input waveform. In this mode, the user specifies the input file, output file, time interval, and either a scale factor or a target duration.

A single-edit, scale-factor-based example is provided as follows:

\begin{verbatim}
python main.py examples/test.wav --output examples/output.wav \
    --t1 0.20 --t2 0.40 --scale 1.5
\end{verbatim}

This command stretches the region from \(0.20\) s to \(0.40\) s by a factor of \(1.5\).

Alternatively, a target duration may be specified directly:

\begin{verbatim}
python main.py examples/test.wav --output examples/output.wav \
    --t1 0.20 --t2 0.40 --target-duration 0.50
\end{verbatim}

In this case, the selected segment is modified so that its output duration is \(0.50\) s.

\subsection{Multi-Edit Usage}

Multi-edit mode allows multiple edits to be applied to the same input file using a CSV specification.

An example of multi-edit mode is provided below:

\begin{verbatim}
python main.py examples/test.wav --output examples/output.wav \
    --csv examples/input.csv
\end{verbatim}

The CSV file must include the columns \texttt{t1} and \texttt{t2}, along with at least one of \texttt{scale} or \texttt{target\_duration}. Each row must specify exactly one of these two duration-modification options.

An example CSV file is provided as:

\begin{verbatim}
t1,t2,scale,target_duration
0.20,0.40,1.5,
0.60,0.80,,0.30
\end{verbatim}

In this example, the first segment is stretched by a factor of \(1.5\), while the second segment is modified such that its new duration is \(0.30\) s.

\subsection{Multi-edit Timing Convention and Validation}

A key design choice in batch mode is that edit times are interpreted relative to the original input-file timeline, rather than the progressively modified output timeline as edits are made. Therefore, if an earlier edit changes the duration of one region, later edit times are still understood as referring to their intended locations by the user in the original signal.

For ease of use and simplicity, overlapping edits in the original timeline are not supported. For example, the following CSV specification is invalid:

\begin{verbatim}
t1,t2,scale
0.20,0.50,1.2
0.40,0.70,0.8
\end{verbatim}

due to the second edit overlapping the first in the original signal.

\subsection{Evaluation and Testing}

In addition to the main processing pipeline, the repository includes a separate evaluation script for comparing original and modified audio outputs. Basic evaluation may be run with:

\begin{verbatim}
python scripts/evaluate.py --original examples/test.wav \
    --modified examples/output.wav --report-dir reports/eval
\end{verbatim}

This generates waveform comparisons, spectrogram visualizations, and a basic metrics report, and writes them to \texttt{reports/eval}. Region-focused evaluation is also supported by specifying the edited interval and a margin around it.

The project also includes an automated test suite, which may be run with:

\begin{verbatim}
pytest
\end{verbatim}

These tests provide some unit test coverage of key framework functionality, including segment extraction, stitching behavior, CSV parsing, batch processing, target-duration handling, remapping of batch edit timelines, and rejection of overlapping edits.

\section{Duration Modification Algorithm}

The core duration-modification method was chosen as a time-domain Pitch-Synchronous Overlap and Add (TD-PSOLA) approach. The TD-PSOLA duration-modification algorithm used in this project was self-implemented via \texttt{psola.py}, using standard Python libraries for array operations and autocorrelation support rather than an external PSOLA codebase. PSOLA is commonly used for speech time-scale modification because it changes segment duration by reusing short waveform regions aligned to the local pitch structure, which can better preserve speech characteristics than standard global resampling.

In \texttt{psola.py}, duration scaling begins by detecting approximately pitch-synchronous marks within the selected segment. This is done using an autocorrelation-based analysis over short frames. For each frame, the algorithm searches for a plausible pitch period within a voiced-speech range, and refines the corresponding mark to a nearby strong waveform extremum. This step is intended to align synthesis more closely to the local waveform structure.

After pitch marks are detected, local pitch periods are estimated from the spacing between neighboring marks. These periods are then used to construct synthesis locations for the output segment. For duration expansion, output pitch-mark positions are placed farther apart, causing waveform grains to be overlap-added over a longer total interval. For duration compression, output positions are placed closer together while source pitch marks are traversed more aggressively, effectively shortening the segment.

The waveform grains used for synthesis are extracted around each detected pitch mark using neighboring pitch marks to define the grain boundaries. Each grain is windowed with a Hann window before synthesis. The modified output is then generated by overlap-adding these grains at the new synthesis positions. A corresponding synthesis-weight array is accumulated and used for normalization so that overlapping regions are averaged appropriately rather than summed without correction.

The implementation also includes several guardrails. If too few pitch marks are detected, which may occur for very short or poorly voiced segments, the system falls back to simple linear interpolation resampling rather than forcing PSOLA on a poor input. After synthesis, the output is normalized if necessary to prevent excessive peak amplitude, and a small DC offset is removed.

Within the broader framework described earlier, this TD-PSOLA module is called through the duration-scaling interface. The extracted waveform segment and requested scale factor are passed into the algorithm, and the resulting modified segment is then returned for stitching back into the original waveform. For target-duration edits, the desired duration is converted externally into an equivalent scale factor so that both user-facing edit modes share the same underlying processing method.

This implementation should be viewed as an introductory TD-PSOLA realization rather than a fully optimized duration modification system. While it successfully performs duration modification on voiced speech segments, its quality depends strongly on pitch-mark accuracy and on the characteristics of the selected segment. Imperfect pitch-mark placement, mixed voiced/unvoiced content, or difficult segment boundaries can lead to audible quirks such as roughness, discontinuities, or a warbled quality in the modified output.

\section{Evaluation Methodology}

Evaluation of the project focused on both software correctness and output quality. Since the project is both a signal-processing system and a software framework, it was necessary to assess not only whether the edited speech exhibited the intended duration change, but also whether the implementation behaved correctly under different edit specifications and processing scenarios.

From a software engineering perspective, the framework was tested using automated tests covering key data-pipeline behavior, including waveform loading, segment extraction, stitching, CSV-based batch edit parsing, target-duration handling, remapping of later multi-edits relative to the original input timeline, and rejection of overlapping edit requests. These tests were used to verify the robustness of the framework independently of the final speech-modification quality.

From a signal-analysis perspective, an evaluation script was developed to compare original and modified audio files. At the full-signal level, this script generates an overlaid waveform comparison, separate spectrograms for the original and modified signals, and a textual metrics summary. The full-signal metrics include sample rate, number of samples, total duration, duration ratio between modified and original audio, RMS amplitude, and peak absolute amplitude.

In addition to full-signal analysis, the script supports region-focused evaluation of a selected edited interval. When the edited segment boundaries are provided, the script extracts the specified time region for both input and output, and generates a regional waveform comparison around that interval. It also generates region-specific spectrograms for both the original and modified signals and writes a separate region metrics file containing the same information in the full-waveform metrics file, only for the specified region.

These outputs were used to evaluate whether the selected segment had undergone the intended duration modification, and to inspect how the spectral and waveform structure changed after processing.

Finally, qualitative listening evaluation was used alongside the generated plots and metrics. Since the project goal is perceptual duration modification of speech rather than exact waveform reconstruction, listening-based assessment was used to judge intelligibility, preservation of speaker character, and the presence of audible byproducts such as warbling or discontinuities around the affected area. Together, these software tests, visual analyses, summary metrics, and listening observations formed the basis of the evaluation.

\section{Results}

The final result successfully implemented an end-to-end workflow for user-directed duration modification of selected speech segments. The framework was able to load an input mono WAV file, accept an edit request, extract the specified segment, apply TD-PSOLA-based duration modification, reconstruct the waveform, and write the edited result to an output file.

The primary demonstration case used throughout development involved a speech recording of a woman reading a narrative passage. The input was composed of the following sentence: 

\begin{quote}
“He had little enough to break or bite, and once, when there was a great famine in the land, he could hardly procure even his daily bread. And as he lay thinking in his bed one night, he sighed and said to his wife, ‘what will become of us’.”
\end{quote}

The demonstration command used for this case was:

\begin{verbatim}
python main.py examples/test.wav --output examples/output_test.wav --t1 5 --t2 10 --scale 1.5
\end{verbatim}

This command applies duration expansion to the segment from 5 s to 10 s in the original waveform using a scale factor of 1.5. The modification begins around the portion of the utterance starting with “he could hardly...”, and the selected region is stretched relative to the original signal.

Evaluation of this demonstration case showed that the system performed the requested duration modification correctly at the file level. The original recording had a total duration of 15.22 s, while the modified output had a duration of 17.72 s, corresponding to an increase of 2.50 s. This matches the requested edit: the selected region was 5 s long, and scaling that region by a factor of 1.5 increases its duration to 7.5 s, producing a net increase of 2.5 s in the full output waveform.

Signal-level summary statistics also remained stable after modification. The RMS amplitude changed only slightly, from 0.0734 in the original signal to 0.0714 in the modified signal, and the peak absolute amplitude remained nearly unchanged, increasing only from 0.7798 to 0.7852. These values indicate that the duration-modification process changed the temporal structure of the selected region without causing large overall amplitude distortion.

Region-focused evaluation around the 5 s to 10 s edit interval also showed clear divergence between the original and modified waveforms, as shown in Figure~1. From a qualitative standpoint, the modified result remained intelligible, and the speech was still recognizable as the same speaker with much of the original tone and inflection preserved. At the same time, the output was not without fault. A listening evaluation indicated audible degradation in the modified segment, particularly at the onset of modification. In addition, a degree of roughness/warbling was introduced by the duration modification process, and is audible in the output audio. These results were noticeable enough that the output clearly sounded modified, although speech characteristics were largely retained, and the speaker's identity was still clearly preserved.

Beyond the main perceptual demonstration, the framework also successfully supported additional functionality such as target-duration editing, CSV-based batch edit input, timeline remapping for multiple edits relative to the original waveform, and rejection of overlapping edit regions. These features were largely proven via unit testing, while the primary qualitative evaluation focused on the single-edit speech demonstration described above, to exhibit the algorithm's effectiveness.

Overall, the project produced a complete and functional duration-editing framework. While the perceptual quality of the modified speech still leaves room for improvement, the final system demonstrates that arbitrary user-specified segment duration changes can be represented, validated, processed, and evaluated within this architecture.

\section{Discussion and Limitations}

In addition to the core duration-modification algorithm, this project resulted in a complete software framework for speech-segment editing, including WAV input/output, edit validation, single-edit and multi-edit workflows, reconstruction logic, automated testing, and evaluation support. From a software-engineering perspective, this broader system design is one of the strongest outcomes of the project.

On the other hand, the perceptual quality of the final edited speech reveals important limitations of the current implementation. Although the system successfully performed the requested duration modification and produced intelligible output, the edited speech was not artifact-free. In the primary demonstration case, the modified segment remained understandable and preserved much of the speaker’s tone and identity, but it also exhibited some roughness and warbling. This indicates that while the TD-PSOLA-based approach was effective in general, it does not yet achieve consistently natural-sounding speech modification.

One likely source of these artifacts is the difficulty of accurate pitch-mark estimation. The implemented method detects approximate pitch-synchronous marks using short-time autocorrelation, and then refines them to nearby waveform extrema. While this approach is reasonable, it is still sensitive to pitch-estimation errors, local waveform irregularities, and transitions between voiced and unvoiced speech. If pitch marks are not aligned well, overlap-add synthesis can introduce discontinuities or unnatural modulation in the edited output, which we see to some extent in the demonstration output.

A second limitation is that the evaluation methodology, while useful, was not exhaustive. The project included waveform comparisons, spectrogram visualizations, and basic signal-level metrics such as duration, RMS amplitude, and peak amplitude, together with qualitative listening-based assessment. These methods were sufficient to verify that the requested duration change occurred and that the broad structure of the speech signal was preserved, but they do not constitute a full speech-processing evaluation. More advanced speech-quality metrics or broader listening studies would be needed to assess naturalness more rigorously.

Future work could improve the system in several ways. More robust pitch-mark detection, smoother boundary handling during reinsertion, and improved treatment of voiced/unvoiced transitions would likely reduce audible artifacts. An additional direction for future work would be support for contour-based duration control, in which the time-scaling applied within a selected segment varies according to a user-specified contour rather than remaining uniform across the entire interval. This would make the system more flexible and could enable more natural or expressive modifications than a single global scaling factor alone. 

Additional evaluation across multiple speech examples and edit configurations would also strengthen confidence in the method. In general however, the architecture presented in this project provides a good foundation for experimenting with refined PSOLA variants or alternative duration-modification techniques in future work.

\section{Conclusion}

This project developed a software framework for arbitrary duration modification of selected segments within a speech waveform. The system supports user-specified edits through either scale-factor-based or target-duration-based input, and provides an end-to-end pipeline including WAV input/output, edit validation, segment extraction, duration modification, waveform reconstruction, batch processing, and evaluation support.

The core duration-modification method selected for the project was a PSOLA-based approach. Within the completed framework, this method serves as the algorithmic mechanism used to lengthen or shorten selected speech regions while attempting to preserve perceptually important characteristics of the original signal.

The final results show that the system successfully performs user-specified duration edits and produces intelligible modified speech, while also revealing the practical challenges of reducing audible artifacts in speech processing. In the primary demonstration case, the requested temporal expansion was achieved correctly at the file level, and the output remained recognizable as the original speaker, although some audible roughness was still present in and around the edited region.

Overall, the project achieved its primary goal of building a functional and extensible duration-modification framework for speech. In addition to satisfying the immediate project objectives, the resulting architecture provides a strong foundation for future refinements to pitch-mark detection, overlap-add synthesis quality, contour-based duration control, and broader perceptual evaluation.
